# artifact-parser

`artifact-parser` is a pluggable framework for turning the JSON artifacts that
data tools emit into typed, validated pydantic models. The framework
(`artifact_parser.core`) is source-agnostic; each **plugin** owns one family of
artifacts and registers itself with a shared registry. The first plugin parses
**dbt-core** artifacts (catalog / manifest / run-results / sources). A separate
developer CLI (`codegen`) regenerates the dbt models from dbt-core's published
JSON schemas.

│       └── dbt/                  #   mirrors artifact_parser.dbt

│           ├── generator.py      #     download + datamodel-codegen + jinja
│           ├── paths.py

`tests/` mirrors `src/` exactly: `tests/artifact_parser/core/`,
`tests/artifact_parser/dbt/`, `tests/codegen/dbt/`.

## Generated vs hand-written

This boundary is load-bearing and now **physical**: everything generated lives
under `artifact_parser/dbt/generated/`. **Never hand-edit a file in there** —
the next `codegen` run clobbers it.

- **Generated by `codegen dbt`** (all under `dbt/generated/`): `models/**`,
  `version_map.py`, `parser.py`, and the package `__init__.py`. Models come from
  `datamodel-codegen`; the version map and parser come from Jinja templates under
  `codegen/dbt/templates/`.
- **Hand-written**: all of `core/`, the dbt `utils.py` / `plugin.py` /
  `__init__.py`, and all of `codegen/`.

The whole `generated/` directory is safe to delete and rebuild:

```
rm -rf src/artifact_parser/dbt/generated
uv run codegen dbt --skip-download   # or drop the flag to pull fresh schemas
```

While it is absent, `import artifact_parser` still works — the dbt plugin just
doesn't register (you get a warning) — so the codegen CLI it needs to rebuild
itself imports cleanly. To change generated output, edit the template or the spec
(`codegen/dbt/artifact_spec.py`), then run `task codegen`.

**Invariant — generated models use `extra="ignore"`, never `"forbid"`.** dbt's
published schemas set `additionalProperties: false`, so `datamodel-codegen` emits
`extra="forbid"`. But real artifacts are a *superset* of the published schema (a
real dbt 1.11 `manifest.json` carries macro `config` fields the v12 schema omits)
— a strict model rejects them outright (hundreds of validation errors). So
`generator.generate_model` runs a post-step, `relax_extra_policy`, that rewrites
`extra='forbid'` → `extra='ignore'` in the raw codegen output (single quotes —
it runs *before* `task format` normalises them). Don't undo this; the round-trip
tests in `tests/artifact_parser/dbt/test_roundtrip.py` (real artifacts in
`tests/data/`) guard it.

## Workflows

Day-to-day work is driven by `task` (root `Taskfile.yml`).

| Goal                                          | Task                  | Slash command |
|-----------------------------------------------|-----------------------|---------------|
| Sync the uv environment                       | `task install`        | —             |
| Format + autofix                              | `task format`         | —             |
| Lint (format-check + ruff)                    | `task lint`           | —             |
| Run tests at 100% coverage                    | `task test`           | `/test`       |
| Regenerate dbt models from cached schemas     | `task codegen:regen`  | `/codegen`    |
| Regenerate + pull fresh dbt-core schemas      | `task codegen:fresh`  | —             |
| Regenerate only if a schema changed (gated)   | `task codegen:check`  | —             |

`task --list` shows everything. `task codegen -- manifest v12` scopes a run to a
single artifact/version.

## Packaging extras

The base install ships only `core`. Pick plugins via extras:

- `pip install artifact-parser[dbt]` — the dbt-core parser.
- `pip install artifact-parser[codegen]` — the model codegen CLI (`typer`,
  `jinja2`, `datamodel-code-generator`). Maintainers only.
- `pip install artifact-parser[codegen,dbt]` — both. This is what the `dev`
  dependency group installs.

## Conventions

- Python: `uv run ruff format && uv run ruff check` must pass. 100% test coverage
  (`task test`). Generated dbt code is omitted from coverage; the dispatch logic
  (`_parse` / `_parse_exact`) is covered through a representative sample of the
  public `parse_*` API.
- No relative imports. All imports at module top (`ruff PLC0415` enforces this).
- One class per file (exception: multiple exception classes may share one file).
  No nested functions/classes.
- Specific exception types from `core/exceptions.py` — never bare `except:`.
- No backward-compat shims unless explicitly asked.
- DRY in tests — share fixtures via `tests/conftest.py`.

## Agentic setup

- `parser-author` agent (`memory: project`) — owns `src/` and `tests/`.
- Slash commands: `/test`, `/codegen`.
- Hooks: `block-secrets.sh` (PreToolUse, denies secret-file access),
  `post-edit-check.sh` (PostToolUse, ruff-checks edited `.py` files).
- CI (`.github/workflows/`): `ci.yml` (lint + 100%-coverage tests on Python
  3.10–3.13 on Linux, plus a codegen-in-sync gate that fails if the committed
  `generated/` drifts from a fresh regen); `release.yml` (build + 100%-coverage
  gate, then PyPI Trusted Publishing on a published GitHub Release, or TestPyPI
  via manual dispatch).

## External docs via context7 MCP

`context7` is configured in `.mcp.json`. Use it to pull up-to-date docs for
`pydantic`, `typer`, `datamodel-code-generator`, and `jinja2` before writing
non-trivial integration code. Prefer context7 over guessing from training data
when library behaviour matters.
