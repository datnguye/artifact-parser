You own `src/` and `tests/`. The project is a pluggable framework for parsing
data-tool artifacts into typed pydantic models. Two packages live under `src/`:

- `artifact_parser/` — the shippable library. `core/` is the source-agnostic
  framework (the `BaseArtifactModel` root, the `ArtifactParser` plugin protocol,
  and the `ParserRegistry`). `dbt/` is the first plugin: it parses dbt-core
  artifacts (catalog / manifest / run-results / sources) and registers itself on
  import.
- `codegen/` — a developer CLI (typer) that regenerates the dbt models from
  dbt-core's published JSON schemas. It mirrors the runtime layout: `codegen.dbt`
  owns the dbt generation.

## Generated vs hand-written

This boundary is load-bearing — respect it:

- **Generated by `codegen dbt`** (never hand-edit) — all under
  `artifact_parser/dbt/generated/`: `models/**`, `version_map.py`, `parser.py`,
  and the package `__init__.py`. The models come from `datamodel-codegen`; the
  version map and parser come from Jinja templates under `codegen/dbt/templates/`.
  The whole `generated/` dir is droppable and rebuildable.
- **Hand-written**: everything in `core/`, the dbt `utils.py` / `plugin.py` /
  `__init__.py`, and all of `codegen/`.

To change generated output, edit the **template or the spec**
(`codegen/dbt/artifact_spec.py`), then run `task codegen` (or
`uv run codegen dbt`). Never patch a generated file in place — the next codegen
run will clobber it.

**Invariant — generated models use `extra="ignore"`, never `"forbid"`.** Real
artifacts are a superset of dbt's published schema (e.g. a real `manifest.json`
carries macro `config` fields the v12 schema omits), so a strict model rejects
them. `generator.generate_model` post-processes its output via
`relax_extra_policy` to flip `extra='forbid'` → `extra='ignore'`. Don't undo it;
`tests/artifact_parser/dbt/test_roundtrip.py` (real artifacts in `tests/data/`)
guards it.

## Adding a new dbt schema version

1. Versions are discovered, not hand-listed: run `task codegen:fresh -- --discover`
   (or `uv run codegen dbt --discover`) to probe `schemas.getdbt.com`, refresh
   `codegen/dbt/versions.py`, and regenerate. For a single known version,
   `task codegen:fresh -- <type> <version>`.
2. Add a minimal fixture in `tests/conftest.py` and cover the new version's
   parser. For populated, real-world coverage, drop an artifact in `tests/data/`
   and assert against it in `tests/artifact_parser/dbt/test_roundtrip.py`.
3. `task test` — keep coverage at 100%.

## Adding a whole new parser plugin

The framework is built for this. Mirror the dbt plugin:

1. `src/artifact_parser/<tool>/` with models on `BaseArtifactModel`.
2. Implement `ArtifactParser` (`name`, `can_parse`, `parse`) in `plugin.py`.
3. Register it in that package's `__init__.py` and import the package from
   `artifact_parser/__init__.py`.
4. If it needs codegen, add a `codegen/<tool>/` sibling.

## Workflow

1. Read the relevant files under `src/`.
2. Make the change (template/spec for generated code; the file itself otherwise).
3. `uv run ruff format . && uv run ruff check .`.
4. `task test` (pytest at 100% coverage).
5. Add tests before reporting done.

## Conventions

- Follow the user's global Python rules: no relative imports, all imports at the
  top of the file, one class per file (exception: multiple exception classes may
  share one file), no nested functions/classes.
- Use specific exception types from `core/exceptions.py`, never bare `except:`.
- DRY in tests — share fixtures via `tests/conftest.py`.
- `tests/` mirrors `src/` directory-for-directory.
- No backward-compat shims unless explicitly asked.
